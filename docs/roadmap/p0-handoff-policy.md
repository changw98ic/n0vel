# P0 缺陷处理规则（Codex）

## 目标
- 将所有阻断性问题定义为 P0，不允许在主干通过或向上版本推进。

## 生效范围
- 本规则覆盖 W0 至 W1 的阻断性缺陷闭环，默认应用于所有 CI 失败、发布前关键安全与数据完整性问题。

## 处理分类

- **P0（阻断）**：测试、CI、发布门禁失败、核心数据安全缺陷
- **P1（高）**：关键体验缺陷、运行治理缺陷、可观测性缺失
- **P2（优化）**：可维护性与一致性优化

## 触发规则
- P0 触发条件：CI 阻断失败、密钥或敏感配置失效、发布前脚本/门禁未达标、或影响主链路可交付性的缺陷。
- 触发入口：PR 检查失败、主分支回归失败、发布演练缺陷发现、或线下回归评审登记。

## 关闭条件
- P0 问题：回归测试通过且通过人审后 24 小时内完成复测
- P1 问题：有明确 owner 与预计修复窗口
- P2 问题：加入下周计划并保留风险说明

## W1 建议阻断测试清单

按阻断优先级执行：

1. `test/app_settings_storage_io_test.dart`  
   - 加密设置加载/存储失败路径（`fileReadFailed/fileWriteFailed`）  
   - 新增（已落库）：密文损坏、密钥不匹配复用失败场景

2. `test/app_settings_store_test.dart`  
   - 连通性校验、路由分发、保存失败状态

3. `test/app_llm_*` 与 `test/story_generation_*`  
   - 失败分类与重试边界、生成流程关键失败传播

4. `test/workbench_shell_test.dart` 与设置页/主流程 UI 测试  
   - 主链路（设置保存/连接提示/异常显示）不得出现阻塞性回归

执行要求：
- P0 回归缺口一旦发现，先补齐对应回归用例再进入同类修复。
- P0 关闭必须附 `flutter test` + 对应回归命令（至少覆盖该文件或标签）与重审时间戳。

## 复测与证据要求（P0）
1. **修复前复现**：附上最小复现路径与环境说明（分支、参数、时间戳）。
2. **修复说明**：记录修复范围、影响面、回归风险。
3. **验证**：
   - 触发 `mvp-blocking-checks` 全量通过（Analyze / Test / Docs Check）。
   - 关键回归用例至少新增一条与问题场景对应的验证条目。
4. **人审签核**：复测完成后由 owner 与复核人二人确认通过后关闭。

## 时限与升级
- P0 问题应在 24 小时内进入修复分支；超时超过 24 小时需升级到里程碑 owner。
- 未在 48 小时内形成可回归修复方案时，进入发布冻结并触发回滚/延期流程。

## 停更与发布冻结规则
- 触发条件：未解决 P0>0 时禁止提测与发布冻结解除。
- 解除条件：满足关闭条件、补齐证据、并在 `mvp-blocking-checks` 全部通过后才可解冻。
