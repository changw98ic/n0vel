# UI 设计稿标准与偏好

本文档用于固定小说创作工作台 MVP 的 UI 设计标准、视觉偏好、组件约束和页面优先级。

文档受众：

- 设计师
- 前端 / Flutter 实现开发
- 产品与评审人员

适用范围：

- MVP
- Windows 桌面端
- 高保真设计稿前的 UI Foundation 定义

不适用范围：

- 官网 / 宣传页完整视觉规范
- 支付、市场、云同步等非 MVP 模块

## 1. 设计总原则

产品整体定位固定为：

- `专业写作工作台`
- `本地优先`
- `长时创作友好`
- `AI 次级但随手可达`

视觉方向固定为：

- 编辑器专业感
- 中高密度
- 浅色默认，暗色可切换
- 柔和卡片 + 清晰边界
- 关键动作有适度叙事反馈

禁止偏移方向：

- 不做聊天应用感
- 不做炫技型 AI 控制台
- 不做宣传页式文学海报
- 不做大面积留白的营销排版

视觉关键词固定为：

- 克制
- 可靠
- 安静
- 长期使用不疲劳
- 带一点文学温度

## 2. 画板与布局基线

高保真设计稿默认画板：

- 主画板：`1440 x 1024`
- 兼容画板：`1280 x 960`

应用壳层布局固定改为：

- 左侧自动隐藏 `menu drawer` 把手
- 顶部 breadcrumb 或轻页眉
- 中部主工作区
- 右侧工具条 / 工具窗口
- 底部状态栏

推荐宽度基线：

- 左侧 menu handle：18-20
- 左侧 drawer 展开态：220-240
- 中部主工作区：自适应，最小 640
- 右侧工具条：48
- 右侧工具窗口：300-320
- 底部状态栏：28-30 高

页面边距与栅格：

- 页面外边距：24
- 区块间距：16
- 卡片内部内边距：16
- 小组件内边距：12
- 紧凑列表行高：36-40
- 标准列表行高：44-48

## 3. 色彩标准

主色主题固定为 `Warm Linen`。

| `bg.elevated` | `#FFFDFC` | 弹层 / 浮层 / 强调卡片 |
| `bg.subtle` | `#EEE6DA` | 次级背景、选中行底 |
| `text.primary` | `#2E2925` | 主文本 |

| `border.default` | `#D8CDC0` | 默认边框 |
| `border.strong` | `#B7AA9A` | 强调边框 |
| `accent.primary` | `#51624D` | 主强调色，按钮 / 选中 / 激活 |

### 3.3 色彩规则

- 强调色只用于：
  - 当前激活项
  - 主按钮
  - 链接态
  - 关键状态反馈
- 页面中不可同时出现多个高饱和强调色争抢注意力。
- 危险色只用于删除、拒绝、严重错误。
- 写作工作台正文区优先保持低干扰，不使用大面积强调底色。

## 4. 字体标准

字体固定：

- `Inter`：标题、模块标题、按钮、标签、导航
- `Geist`：正文编辑、长说明、问卷描述、日志正文

### 4.1 字号层级

| Token | 大小 | 行高 | 字体 |
| --- | --- | --- | --- |
| `type.pageTitle` | 24 | 32 | Inter |
| `type.panelTitle` | 18 | 26 | Inter |
| `type.sectionTitle` | 16 | 24 | Inter |
| `type.body` | 15 | 24 | Geist |
| `type.bodyCompact` | 14 | 22 | Geist |
| `type.label` | 13 | 18 | Inter |
| `type.caption` | 12 | 18 | Inter |
| `type.codeLike` | 13 | 20 | Geist |

### 4.2 排版规则

- 正文编辑器正文使用 `Geist 15 / 24`。
- 长段落最大宽度建议控制在 68-76 个中文字符内。
- 模块标题不使用全大写。
- 次级说明统一使用 `text.secondary`，不使用更浅的占位色替代正文信息。
- 列表与日志内容优先保证垂直对齐和行距一致。

## 5. 组件标准

组件造型固定为 `Basic Roundness`。

### 5.1 圆角

| 组件 | 圆角 |
| --- | --- |
| 小按钮 / 输入框 / 标签 | 8 |
| 卡片 / 侧栏容器 / 面板 | 12 |
| 弹层 / 对话框 | 14 |

禁止事项：

- 不做胶囊型超大圆角按钮
- 不做圆角值混乱的组件集合

### 5.2 边框

- 常规边框：`1px solid border.default`
- 强调边框：`1px solid border.strong`
- 正文编辑区边框必须弱于右侧 AI 操作分区，不抢正文视线

### 5.3 阴影

| 层级 | 阴影 |
| --- | --- |
| 普通卡片 | 无阴影或极弱阴影 |
| 浮层 / 菜单 | `0 8 24 rgba(0,0,0,0.08)` |
| 对话框 | `0 12 32 rgba(0,0,0,0.12)` |

规则：

- 主工作台尽量靠边框与明度分层，而不是靠厚重阴影。
- 阴影只用于区分漂浮层级，不用来做装饰。

## 6. 组件状态

### 6.1 Button

按钮分为 4 类：

- `Primary`
- `Secondary`
- `Tertiary`
- `Danger`

状态：

- default
- hover
- pressed
- disabled
- loading

规则：

- 主工作台同一区域最多一个 `Primary` 按钮。
- `接受草稿` 使用 `Primary`。
- `拒绝草稿` 使用 `Danger` 或带危险语义的次级按钮。
- `重新叙述`、`局部重写`、`暂停/继续/中止` 使用 `Secondary` 或 `Tertiary`。

### 6.2 Input

输入类组件统一状态：

- default
- focused
- filled
- error
- disabled

规则：

- focused 必须同时体现边框变化和背景/阴影微变化。
- error 态必须带说明文本，不能只靠红边框。

### 6.3 List / Row

统一适用于：

- 项目列表
- 场景树
- 日志列表
- 审计列表
- 版本列表

状态：

- default
- hover
- selected
- disabled

规则：

- selected 只使用一层柔和底色 + 左侧高亮条，不使用高饱和整行染色。
- hover 与 selected 必须可并存且可区分。

## 7. 动效标准

动效策略固定为 `适度叙事型`。

### 7.1 时长

| 场景 | 时长 |
| --- | --- |
| hover / press | 100-150ms |
| 面板展开 / 收起 | 180-220ms |
| 内容替换 / 成功反馈 | 220-300ms |

### 7.2 动效范围

允许出现动效的地方：

- 面板展开与收起
- 模拟运行状态推进
- 草稿接受 / 拒绝反馈
- 局部重写高亮替换

禁止事项：

- 大面积位移
- 缩放弹跳
- 持续循环动画
- 会影响长时间阅读的闪动和脉冲

## 8. 图像与装饰

`Candid Lifestyle Photography` 只允许用于：

- 项目列表空状态
- 首次使用引导
- 风格问卷空状态
- 官网 / 宣传层

禁止用于：

- 写作工作台主界面
- 沙盒监视器
- 设置与 BYOK
- 任何核心表单页

工作台、沙盒监视器、设置页全部以：

- 结构
- 排版
- 层次

为主，不依赖摄影图像做视觉成立。

## 9. 页面优先级与页面级规则

### 9.1 写作工作台

这是唯一必须达到“高完成度高保真”的页面。

固定规则：

- 正文编辑区是视觉主角，占最大面积和最高焦点
- AI 操作区位于右侧次级区，可见但不压过正文
- 接受 / 拒绝 / 局部重写必须独立可见，不得折叠进更多菜单
- 底部模拟日志区必须稳定、低干扰、易扫读

### 9.2 沙盒监视器

固定规则：

- 必须视觉区分三类信息：
  - 发言
  - 意图
  - 裁决
- 不得做成终端风或日志墙
- 状态快照区要安静，不抢日志主区注意力

### 9.3 风格面板

固定规则：

- 问卷入口与 JSON 导入入口必须同级出现
- 中间摘要区像“已定风格卡”，不是代码检查器
- 问卷分组必须强引导、强层级

### 9.4 其他页面

项目列表、角色库、世界观页、版本历史、审计中心、设置页共用同一套壳层：

- 同样的顶部结构
- 同样的卡片和列表规则
- 同样的筛选区与详情区节奏

不允许每页重新发明一套视觉语言。

## 10. UI Foundation 交付项

设计师在正式出稿前必须先产出一页 `UI Foundation`，至少包含：

- 色彩 token
- Typography scale
- Radius / border / shadow
- Button states
- Input states
- Panel
- Card
- Sidebar
- Tab
- Table row
- Log row

MVP 高保真稿至少覆盖：

- 写作工作台
- 沙盒监视器
- 风格面板
- 项目列表
- 设置与 BYOK

## 11. 组件命名建议

组件命名规则固定为：

- `AppShell/*`
- `Panel/*`
- `Editor/*`
- `Simulation/*`
- `Form/*`
- `Feedback/*`

示例：

- `AppShell/MainLayout`
- `Editor/BodyPane`
- `Simulation/RunStatusBar`
- `Form/QuestionGroup`
- `Feedback/InlineError`

## 12. 验收标准

- 只看写作工作台截图，用户应在 3 秒内判断这是专业创作工具，而不是聊天助手或通用 SaaS。
- 连续阅读正文 20 分钟，不因颜色、字重或动效产生明显疲劳。
- 写作工作台中，用户 3 秒内能分辨：
  - 正文主区
  - AI 操作区
  - 模拟状态区
- 沙盒监视器中，用户 5 秒内能分辨：
  - 发言
  - 意图
  - 裁决
- 风格面板中，用户 5 秒内能发现：
  - 问卷入口
  - JSON 导入入口
  - 风格强度控制
- 浅色与暗色都必须保持同一品牌气质，不能一个像写作工具、一个像代码编辑器。

## 13. 实施顺序建议

1. 先做 `UI Foundation`
2. 再做写作工作台高保真稿
3. 再做沙盒监视器高保真稿
4. 再做风格面板高保真稿
5. 最后用统一壳层扩展其余页面
