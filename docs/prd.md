# 最新 PRD

更新日期：2026-05-12

## 1. 产品定位

`n0vel` 是一款面向长篇小说创作的本地优先桌面写作工作台，聚焦项目资料、正文写作、AI 辅助、工作流结束后的用户反馈和导出归档，形成一条可控、可追溯、可恢复的创作主线。

当前版本坚持四条原则：

- 本地优先：项目数据、草稿、运行快照和大部分状态保存在本地。
- 作者主导：AI 只能生成、重写、建议，不能绕过作者直接固化正文。
- 结构化创作：角色、世界观、场景和正文上下文都是长期资产。
- 可恢复：失败时必须说明影响范围，并允许作者继续推进工作。

推广与增长工作围绕同一个定位展开：给长篇小说作者用的本地优先 AI 创作工作台。执行细节不改变当前产品范围，单独沉淀在 [推广与增长路线图](growth-roadmap.md)。

## 2. 目标与非目标

### 2.1 当前目标

- 让作者以“项目”为单位持续维护长篇作品。
- 让 AI 介入保持清晰边界与作者主导。
- 让单场景写作、反馈、修订和导出形成闭环。
- 让模型接入、运行状态和失败原因对作者可见。

### 2.2 明确非目标

- 不提供云同步、多用户协作或在线托管。
- 不内置资产市场、支付或平台交易审核。
- 不把多章全自动编排作为当前主交付。
- 不把聊天界面作为正文创作主界面。

## 3. 范围边界

- 平台形态：Flutter 桌面客户端。
- AI 接入：通过用户自配 provider 与模型端点接入。
- 运行方式：无强制后端依赖，核心编排、状态和持久化可在本地完成。
- UI 稿源：`editor.pen` 是唯一有效 UI 稿。
- 架构细节：实现分层、路由和数据流细节以 [最新架构图](architecture.md) 为准。

## 4. 产品范围

当前版本只保留 5 组核心能力：

| 模块 | 说明 |
| --- | --- |
| 项目与资料 | 管理项目、角色、世界观、场景和参考资料，作为长期创作资产。 |
| 写作主线 | 以写作工作台为正文生产主入口，并提供正文回看与自动快照保护。 |
| AI 运行 | 提供生成、重写、续写、过程记录和失败回看，但不允许 AI 静默改正文。 |
| 反馈与检查 | 在工作流结束后通过用户反馈或用户主动发起一次规则检查来决定写入、修改或重跑。 |
| 设置与导出 | 管理模型配置、密钥、连接测试，以及工程导入导出。 |

## 5. 核心业务流程

### 5.1 建立项目

1. 作者创建或进入一个小说项目。
2. 项目进入工作区后，继续维护角色、世界观、场景、参考资料和正文。
3. 项目切换后，相关页面必须恢复到对应项目上下文。

### 5.2 写作与 AI 辅助

1. 写作工作台始终是正文主入口。
2. 作者可发起生成、重写、续写和模拟。
3. 任何会改动正文的 AI 结果都必须先回到作者确认。
4. 工作流过程必须可回看，失败时必须明确正文是否受影响。

### 5.3 工作流结束与用户反馈

1. 工作流完成后，先向作者展示候选正文和过程摘要。
2. 作者在结束态通过对话交互给出反馈，例如采纳写入、要求改写、补充修改意图，或主动发起一次规则检查。
3. 规则检查由作者主动触发，其结果用于辅助当前轮判断。
4. 只有跨章节、跨资料或当前无法在这一轮收尾中解决的问题，才进入稍后处理。
5. 写入正文后，系统自动生成本地快照作为回退保险。
6. 导出负责把项目沉淀为可迁移、可归档工件。

### 5.4 工作流边界与状态

1. 每次工作流运行是一个独立生命周期，包含草稿、候选生成、反馈、检查、写入、失败、取消和恢复等明确阶段。
2. 中途关闭应用后，作者重新进入时应能看到上次未完成运行的恢复提示，并能选择继续或丢弃。
3. 作者可在反馈后发起重跑；重跑创建新的候选，不覆盖已有候选，直到作者明确写入。
4. 规则检查失败时，结果保留在工作流结束态供作者判断，不自动阻断写入。
5. 同一章节同一时刻只允许一个活跃工作流运行；作者必须对前一个运行做出终态决策后才能启动新运行。

### 5.5 数据隔离与持久化

1. 工作流运行产出的候选正文、规则检查结果和运行日志都限定在该运行和对应章节/项目范围内，不泄漏到其他运行或章节。
2. 只有作者明确采纳的候选正文才能进入正式草稿；未被采纳的候选正文、运行日志和检查结果属于运行临时数据。
3. 项目资料和设置在工作流启动时以快照形式读入，运行期间不受作者中途修改设置的影响。
4. 以下数据必须立即持久化：项目主数据、草稿、自动快照、配置。
5. 以下数据为运行临时数据：未采纳的候选正文、规则检查结果、运行日志。运行结束后作者可选择保留或清理。

## 6. 验收原则

- 正文编辑始终是主工作流，AI 区只能是次级辅助。
- 资料页必须是长期资产管理界面，不能退化成一次性 prompt 填空。
- AI 运行必须有阶段反馈、失败说明、恢复入口和结果回看。
- 问题来源以用户反馈或用户主动发起的规则检查为主。
- 工作流结束态必须支持对话式反馈、写入、改写和规则检查。
- 设置变更必须作用于后续请求，但不能破坏已有项目数据。
- 导出结果必须适合备份和迁移。

## 7. 非功能要求

- 本地持久化：项目、草稿、运行快照、自动快照和配置应可恢复。
- 可观测性：关键链路应有状态、日志或快照证据。
- 可恢复性：失败后要说明“哪里失败、正文是否改动、下一步能做什么”。中途退出后应能恢复到上次未完成的运行状态。
- 数据隔离：运行临时数据与项目主数据严格分离，运行间互不干扰。
- 可维护性：产品继续扩展时，页面、运行时和持久化边界不能重新混成一层。

## 8. 视觉与交互基线

- `editor.pen` 是唯一 UI 真源。
- 整体方向：专业写作工作台。
- 正文编辑区优先级最高，AI 区和摘要区保持次级权重。
- 阅读模式强调低干扰，工作流过程视图强调结构化信息，不做日志墙。
- 工作流结束态应体现“正文预览 + 用户对话 + 可选规则检查”的收尾交互，并在侧栏显示运行状态、恢复提示和规则检查结果。
- 工作流结束态的侧栏状态卡片应覆盖：等待反馈、可恢复草稿、规则检查失败、源内容已变更需确认、已写入完成等关键状态。
- 非阅读类页面尽量共享同一套壳层与视觉语言。
