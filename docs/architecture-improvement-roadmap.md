# 架构改进路线图（修复版）

> 生成日期：2026-05-13  
> 基于：项目实现现状 + 风险优先修复视角
> 目标：把“愿景项”改造为“可执行、可验收、低回归风险”的实施计划

---

## 一、先说明当前偏差（基线对齐）

该项目已有部分能力已接近目标状态（如部分持久化校验、WAL、备份与崩溃恢复）。本次修订不做重复建设，而是把当前“零散能力”收敛为统一机制，并优先修复最危险的风险路径。

| 维度 | 当前状态 | 与原计划关系 | 风险 |
|---|---|---|---|
| 数据持久化失败处理 | 部分实现但不统一，仍有隐式静默路径 | P0#1 方向有效，但实现未全局化 | 高 |
| WAL/备份/恢复 | 已有 WAL 与备份能力，但启动完整性巡检/集中策略不足 | P0#2 方向有效，目标未闭环 | 中 |
| Store 解耦 | 部分改造存在，关键 Store 间仍有直接耦合 | P1#3 仍是关键重构项 | 高 |
| 生成管线参数 | 已有参数外置，但缺统一“运行配置源”与治理策略 | P1#4 需补齐治理层 | 中 |
| Settings 结构 | 已有 mixin/分层基础，仍需统一入口与边界 | P1#5 目标需收敛 | 中 |
| 关系/搜索/统计/弧线 | 存在部分模型/检索能力，但产品闭环不足 | P2 全部为功能闭环补齐 | 中高 |

---

## 优先级定义

| 级别 | 含义 | 响应时间 |
|------|------|----------|
| P0 | 数据安全/系统稳定性 | 立即 |
| P1 | 架构健康/可维护性 | 1-2 周内 |
| P2 | 功能缺失影响核心体验 | 按迭代排期 |
| P3 | 长期规划 | 积压池 |

---

## 风险与痛点修复总原则

1. 先保全现有实现，再统一抽象，避免“大重构一次到位”。
2. 每个风险点都要求：
   - 识别路径（触发条件）
   - 规避路径（技术方案）
   - 观测指标（验收）
3. 任何会改变持久化和事件顺序的改动必须配一条回归检查。
4. 所有 P0 与高风险 P1 在第一阶段完成后再推进新产品功能。

---

## P0 - 数据安全（立即修复）

### 1. 持久化静默失败统一修复（高优先级）

**现状**：存在 `unawaited(_persist())` 或等价的 fire-and-forget 写入路径，失败通常不进入统一可见面。

**修改范围**：
- `lib/app/state/` 下持久化入口
- `lib/app/state/workspace_facades/` 下持久化代理层

**修复动作**：
- [ ] 建立 `PersistenceFailurePolicy`（统一模块）
  - 统一封装：重试次数、退避策略、失败分类（可重试/不可重试）
  - 统一告警输出：将持久化失败映射为可见提示事件
- [ ] 由“直接调用 `unawaited(_persist())`”改为“统一持久化协调服务”
- [ ] 失败事件挂接 UI 通知与日志链路
- [ ] 增加最小化日志：失败路径、重试次数、失败原因、恢复时间

**验收标准**：
- 可复现故障路径下，持久化失败会触发用户可见提示并落日志
- 重试路径至少 1 次成功恢复（可人工复现）
- 所有主要 store 覆盖统一策略，未覆盖项列入下一阶段清单

**对应风险缓解**：
- 消除“静默失败导致用户长期发现不了数据未写入”的隐性风险
- 降低跨 store 的状态漂移风险

---

### 2. 启动完整性与备份策略修复（高优先级）

**现状**：已有 WAL/备份/崩溃恢复相关能力，但缺“启动全链路健康巡检 + 恢复策略集中化”。

**修改范围**：
- `lib/app/state/app_workspace_storage_io.dart`
- `lib/app/state/app_auto_backup.dart`
- 启动流程相关初始化入口

**修复动作**：
- [ ] 建立 `StartupIntegrityCheck`
  - 检查项：journal 状态、schema 版本、关键表一致性、索引状态、最近 checkpoint/快照状态
- [ ] 将 `PRAGMA integrity_check` 纳入启动健康流程（失败走降级恢复分支）
- [ ] 备份服务统一化为 `BackupManager`
  - 统一备份目录索引
  - 自动保留策略（最近 10 个 + 每日快照 30 天）
  - 与恢复界面共享恢复候选项元数据
- [ ] 与现有 crash 恢复页联动：给用户提供“只读检查模式 / 直接恢复模式”

**验收标准**：
- 已知损坏场景下应用能在启动阶段给出明确恢复分支提示
- 恢复动作可追踪到备份元数据（来源、时间、版本）
- 备份成功率与可恢复率可在日志中查询

**对应风险缓解**：
- 减少崩溃后“恢复成功但用户迷路”的体验风险
- 避免恢复策略分散导致的不可预测行为

---

## P1 - 架构健康（1-2 周）

### 3. Store 依赖解耦（高优先级）

**现状**：关键运行链路存在直接导入与同步调用，短期内风险可控，但长期难以维护。

**修改范围**：
- `lib/app/state/story_generation_run_store.dart`
- `lib/app/state/story_generation_store.dart`
- 依赖这些 store 的上层编排层

**修复动作**：
- [ ] 按“先读后写”分两步：
  - 第一步：抽取接口（`StoreContract`）并把跨 Store 访问集中到服务层
  - 第二步：事件化核心只读/轻状态更新
- [ ] 与 EventBus 配套定义最小事件 schema（含版本号）
- [ ] 新增禁止规则：新代码不允许新增 store 间直接 import；现有路径列入迁移清单
- [ ] 建立迁移矩阵：每个 store 的依赖关系与替换进度

**验收标准**：
- 2 周内完成高优先链路迁移，遗留依赖可追踪且有 owner
- 关键行为回归通过（尤其是生成链路和 UI 呈现）
- 新增实现 0 件引入跨 store 直接 import

**对应风险缓解**：
- 降低耦合导致的修改半径过大与回归面扩展
- 为后续单元/并行开发留出稳定边界

---

### 4. 生成管线配置化与可追踪性（中高优先级）

**现状**：部分参数已经外置，但配置源不统一；参数变更缺统一治理口。

**修改范围**：
- `lib/features/story_generation/data/chapter_generation_orchestrator.dart`
- `lib/app/llm/app_llm_request_pool.dart`

**修复动作**：
- [ ] 统一配置对象 `GenerationPipelineConfig` 为运行时入口
- [ ] 引入配置分层：系统默认 > 模型预设 > 项目覆盖 > 会话临时覆盖
- [ ] 把 retry/token/budget/concurrency 列为可观测配置字段（含来源）
- [ ] 配置变更支持回退和快照导出，便于复现问题

**验收标准**：
- 支持切换模型/角色后策略自动适配
- 无硬编码魔法数字散落在 orchestrator 核心流程中
- 同一输入同一配置下结果可复现（按可观测配置快照）

**对应风险缓解**：
- 避免“参数修改只能改代码”导致的发布风险
- 防止参数调整失控与实验不可复现

---

### 5. Settings 结构收敛（中高优先级）

**现状**：已开始采用分离式实现，但聚合入口和职责边界仍可收紧。

**修改范围**：
- `lib/app/state/app_settings_store.dart`
- `lib/app/state/settings/` 目录

**修复动作**：
- [ ] 明确三层职责：读取（query）、校验（validate）、持久化（persist）
- [ ] 去除桥接式“万能透传”接口，保留稳定的设置 facade
- [ ] 将用户可见变更路径压缩为“单一路径写入”

**验收标准**：
- 通过单点设置变更不出现双写
- settings 回归测试覆盖主要高级设置项
- 新建设置项不再新增桥接方法

**对应风险缓解**：
- 降低设置分支失联导致的“页面显示值”和“持久化值”不一致

## P2 - 核心体验补齐（按迭代排期）

### 6. 角色关系持久化（高优先级）

**现状**：关系语义在运行时模型较完整，但缺少可维护持久实体。

**修改范围**：
- `lib/domain/workspace_models.dart`（新增/扩展 `CharacterRelation`）
- `lib/app/state/workspace_facades/`（关系 CRUD）
- `lib/features/characters/`（关系编辑面板）
- 生成管线上下文注入点

**修复动作**：
- [ ] 增加 `CharacterRelation` 表（含版本与更新时间）
- [ ] 支持添加/编辑/删除关系并持久化
- [ ] 生成阶段读取持久关系作为 context enrichment 输入
- [ ] 关系变更建议回写，默认走用户确认流程
- [ ] 增加数据一致性校验（重复关系、反向关系、孤儿 ID）

**验收标准**：
- 关系可跨会话持久存在
- 生成结果与关系上下文对齐率可追踪
- 用户可回看和手动修正关系历史

**对应风险缓解**：
- 缓解“AI 关系漂移”与“用户修改丢失”的核心体验痛点

---

### 7. 全文搜索与跨章节定位（中高优先级）

**现状**：检索组件已具备基础能力，但缺完整用户体验闭环。

**修改范围**：
- `lib/app/rag/` 相关检索与索引模块
- 搜索入口与结果落地页

**修复动作**：
- [ ] 建立统一搜索面板（关键词、角色过滤、章节范围）
- [ ] 将检索结果与场景/章节双向跳转打通
- [ ] 加入命中高亮、分页、排序策略（标题/正文/语义权重）
- [ ] 增量索引策略：写入后异步更新索引，失败重试可视化

**验收标准**：
- 在 `100 章 × 3000 字/章` 的本地数据规模下，搜索返回 1 秒内可操作结果
- 高亮定位跳转准确率不低于 95%
- 索引更新失败有重试与告警

**对应风险缓解**：
- 解决长文档下“找不到上下文”的写作效率问题

---

### 8. 写作统计与目标管理（中优先级）

**现状**：统计能力尚未形成统一实体模型与看板。

**修改范围**：
- `lib/app/state/` 状态层
- 统计/目标相关页面

**修复动作**：
- [ ] 引入 `WritingStats` 统一聚合模型
- [ ] 建表保存日/周/项目级统计（字数、产出章节、目标达成）
- [ ] 写作事件写入即更新统计，避免全量重算
- [ ] 提供目标提醒与趋势图

**验收标准**：
- 统计值与内容字数口径一致
- 写作事件到指标更新延迟 < 3s
- 目标达成提醒可配置且可关闭

**对应风险缓解**：
- 减少“看着像在写但无量化反馈”的流失风险

---

### 9. 大纲/情节线可视化（中优先级）

**现状**：叙事弧线相关 domain 存在，但缺完整持久化和可交互视图。

**修改范围**：
- `lib/features/story_generation/` 领域与页面层
- 工作区 schema 与持久化层

**修复动作**：
- [ ] 将 `StoryArc` 纳入工作区对象与版本管理
- [ ] 增加时间线视图 + 悬空伏笔告警
- [ ] 允许拖拽调整章节顺序并回写
- [ ] 视觉层与数据层保持单向同步，支持撤销

**验收标准**：
- 从大纲更新到弧线图更新延迟可观测且可回退
- 伏笔追踪状态可见、可手动修正
- 排序/拖拽结果可恢复到场景顺序

**对应风险缓解**：
- 避免长篇项目中章节、弧线与情节线断裂导致的逻辑断点

## P3 - 长期规划

### 10. 写作风格学习
- 目标：从用户历史文稿提取风格分布，并在生成前可插拔应用。
- 风险对策：优先做可解释特征展示（词汇/句式/节奏）再进入自动化应用。

### 11. 实时写作辅助
- 目标：Inline 建议、下一句推荐、模板插入。
- 风险对策：先引入“可撤回建议列表”而非自动落盘。

### 12. 批注与修订系统
- 目标：修订来源、审批、版本 diff。
- 风险对策：先搭基础的块级变更日志与来源追踪。

### 13. 素材/模板库
- 目标：跨项目共享模板与素材。
- 风险对策：与关系/弧线视图统一元数据标签体系。
